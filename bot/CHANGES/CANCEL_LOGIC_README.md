# Система универсальной отмены — актуальная реализация

## Обзор

Улучшение клиентского пути для менеджера:
раньше не было кнопки отменить, была только кнопка "Отмена", которая вела в главное меню.
Из-за того, что добавились новые меню пришлось внедрить "Отмену" для того, чтобы интерфейсом было удобно пользоваться.
Добавлены кнопки отмены для сотрудников где это необходимо (При регистрации - нет, при изменении данных - да)
Создана универсальная система отмены, которая анализирует контекст пользователя и автоматически определяет, куда его вернуть при нажатии кнопки "Отмена".
Более лаконичные и понятные текстовыки для менеджера.
Переработаны все меню менеджера.

Изменены клавиатуры. Раньше клавиатуры были встроенные, клавиатурные.
Сейчас везде используются инлайн кнопки, которые позволяют более гибко управлять контекстом и действиями пользователя.

Добавлены логирование действий отмены, чтобы отслеживать, какие действия выполняются и в каком контексте.

## Архитектура

### Основные компоненты

1. **`_determine_cancel_action(user_id, current_state)`** - определяет тип действия отмены
2. **`_execute_cancel_action(callback, state, action_type, additional_data)`** - выполняет соответствующее действие
3. **Специализированные обработчики** - для каждого типа отмены

### Типы действий отмены

- `manager_menu` - возврат в меню менеджера
- `employee_menu` - возврат в меню сотрудника  
- `new_user` - возврат в главное меню для новых пользователей
- `team_settings` - возврат к настройкам команды (используется «Назад» в настройках времени)

## Логика определения действия

### Для менеджеров:
- Всегда → возврат в главное меню менеджера

### Для сотрудников:
- Всегда → возврат в меню сотрудника

### Для новых пользователей:
- Всегда → возврат в главное меню

## Использование

### Основной обработчик
```python
@router.callback_query(F.data == "cancel_action")
async def universal_cancel_handler(callback: CallbackQuery, state: FSMContext):
    # Автоматически определяет и выполняет нужное действие
```

### Специализированные обработчики
```python
@router.callback_query(F.data == "cancel_team_edit")
async def cancel_team_edit_handler(callback: CallbackQuery, state: FSMContext):
    # Возврат в меню менеджера
```

## Преимущества

1. **Единая точка входа** - все отмены обрабатываются централизованно
2. **Автоматическое определение контекста** - не нужно указывать куда возвращаться
3. **Декомпозиция** - логика разбита на понятные функции
4. **Расширяемость** - легко добавить новые типы отмены
5. **Переиспользование** - функции можно использовать в других местах
6. **Подробное логирование** - фиксируем состояние, роль, направление возврата

## Примеры сценариев

### Сценарий 1: Менеджер в настройках времени
1. Пользователь нажимает "Отмена"
2. Система определяет: менеджер + состояние TeamTimeSettings
3. Возврат к настройкам команды (единый экран с шаблоном настроек)

### Сценарий 2: Менеджер в обычном меню
1. Пользователь нажимает "Отмена"
2. Система определяет: менеджер + обычное состояние
3. Возврат в меню менеджера

### Сценарий 3: Сотрудник
1. Пользователь нажимает "Отмена"
2. Система определяет: сотрудник
3. Возврат в меню сотрудника

## Логирование

Система ведет подробные логи всех действий отмены:

### Примеры логов:

```

Отмена в настройках времени для менеджера 123456 (команда: Тестовая) - возврат в главное меню
Отмена выполнена для пользователя 123456 - действие: manager_menu
```

```
Нажата кнопка 'Отмена' пользователем 789012 в состоянии: None
Отмена для сотрудника 789012 (Иван Ив) - возврат в меню сотрудника
Отмена выполнена для пользователя 789012 - действие: employee_menu
```
